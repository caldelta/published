01
Định Nghĩa — Mono và CoreCLR là gì?
Legacy Runtime
Mono Runtime
Mono là implementation open-source của .NET Framework, được Novell (sau là Xamarin/Microsoft) phát triển từ 2001. Unity adopt Mono từ 2005 để chạy C# script trên đa nền tảng. Mono không phải do Microsoft tạo ra — đây là bản port độc lập của CLI specification.
Nguồn gốcNovell → Xamarin → Microsoft
Năm ra đời2001 (dùng trong Unity từ 2005)
.NET tương đương.NET Framework 4.x
C# versionC# 9 (giới hạn)
Scripting BackendMono hoặc IL2CPP
Trạng tháiLegacy — Đang bị thay thế
Modern Runtime
.NET CoreCLR
CoreCLR là Common Language Runtime của .NET Core / .NET 5+ — được Microsoft viết lại hoàn toàn từ đầu, thiết kế cho hiệu năng cao, cross-platform, và modern workloads. Đây là runtime chính thức của Microsoft, không phải port. Unity đang adopt CoreCLR để thay thế Mono trong các phiên bản mới.
Nguồn gốcMicrosoft (viết lại từ đầu)
Năm ra đời2016 (.NET Core 1.0)
.NET tương đương.NET 8 LTS
C# versionC# 12 (đầy đủ)
Scripting BackendCoreCLR (JIT) hoặc IL2CPP
Trạng tháiActive — Future of Unity
ℹ
IL2CPP vẫn là option riêng
IL2CPP không phải runtime — đây là AOT compiler chuyển IL bytecode thành C++ code trước khi build. Mono và CoreCLR đều là JIT runtime. IL2CPP là lựa chọn tách biệt, thường dùng cho iOS và console builds. Bài này tập trung vào Mono JIT vs CoreCLR JIT khi chạy trong Editor và trên thiết bị.

Kiến Trúc — Tầng Phía Dưới

↓ compile to IL bytecode
Mono JIT Compiler (Mini JIT)
↓ translate to native code
Mono GC (Boehm GC — non-moving)
↓ memory management
Mono Class Library (mscorlib.dll)
↓ P/Invoke / Embedding API
Native C++ (Unity Engine Core)
Mono dùng Boehm GC — conservative, non-moving garbage collector. Không compact heap → fragmentation theo thời gian. JIT compiler (Mini JIT) đơn giản, ít optimization.
CoreCLR Runtime Stack
C# Code (UnityEngine.dll, user scripts)
↓ compile to IL bytecode
RyuJIT Compiler (tiered compilation)
↓ translate to native code (optimized)
CoreCLR GC (moving, generational)
↓ memory management
.NET BCL (System.*, Microsoft.*)
↓ P/Invoke / Hosting API
Native C++ (Unity Engine Core)
CoreCLR dùng RyuJIT với tiered compilation — method được compile ở tier 0 (nhanh) rồi recompile ở tier 1 (optimized) nếu được gọi nhiều. GC là moving, generational (Gen0/1/2 + LOH).
Thành Phần Mono CoreCLR
JIT Compiler Mini JIT — đơn giản, ít optimization pass, không hỗ trợ SIMD tốt RyuJIT — tiered compilation, SIMD, vectorization, loop unrolling
Garbage Collector Boehm GC — conservative, non-moving, single generation. Dễ fragmentation heap. Moving, generational GC (Gen0/Gen1/Gen2/LOH). Compact heap, ít fragmentation hơn.
Tiered Compilation Không có — mỗi method compile 1 lần duy nhất Có — Tier 0 (nhanh) → Tier 1 (optimized) dựa trên hot path
SIMD / Vectorization Hạn chế — partial SIMD, không full AVX2/AVX-512 Đầy đủ — System.Runtime.Intrinsics, auto-vectorization
Class Library Mono mscorlib — subset của .NET Framework, một số API không có hoặc khác behavior .NET 8 BCL đầy đủ — System.*, Microsoft.*, tất cả modern API
Thread / Async ThreadPool cơ bản, async/await support nhưng không optimized ThreadPool redesign, ValueTask, IAsyncEnumerable, channel-based concurrency
Native Interop P/Invoke — hoạt động nhưng overhead cao hơn, không có source-generated marshalling P/Invoke + LibraryImport (source-generated, zero-overhead marshalling)
Startup Time Nhanh hơn — JIT đơn giản, khởi động nhanh Chậm hơn lần đầu — tiered JIT cần warm-up, sau đó nhanh hơn nhiều
Debug Support Mono soft debugger — chậm hơn, ít features Full .NET debugger — nhanh hơn, hot reload, edit-and-continue
03
Quản Lý Memory
Mono — Boehm GC
Conservative, Non-Moving GC
Conservative scanning — GC scan tất cả memory để tìm pointer, kể cả stack. Không biết chắc cái gì là pointer, cái gì là integer → có thể giữ object không cần thiết.
Non-moving — object không bao giờ được di chuyển trong heap. Không compact → heap bị fragmented theo thời gian, đặc biệt khi alloc/free nhiều object nhỏ.
Stop-the-world — khi GC chạy, toàn bộ thread bị dừng. Gây ra hiện tượng frame spike trong game.
Không có generation — tất cả object được xử lý như nhau, không phân biệt short-lived vs long-lived object.
GC.Collect() là cần thiết — dev phải quản lý bằng tay để tránh spike. Unity cung cấp Incremental GC nhưng chỉ giảm, không loại bỏ vấn đề.
CoreCLR — Moving Generational GC
Generational, Compacting GC
Precise tracking — GC biết chính xác mọi reference. Không conservative scan → không giữ object sai, memory usage thấp hơn.
Moving / Compacting — sau GC, object được compact lại. Loại bỏ fragmentation, cải thiện cache locality của CPU.
Generational — Gen0 (short-lived, GC thường xuyên + nhanh), Gen1 (buffer), Gen2 (long-lived, GC ít + chậm). LOH cho object lớn (>85KB).
Background GC — Gen2 và LOH có thể collect trên background thread, không block main thread. Gen0/Gen1 vẫn stop-the-world nhưng rất nhanh.
Span<T> / Memory<T> native support — runtime được thiết kế để work với zero-allocation patterns, giảm GC pressure đáng kể.
⚠
GC Spike vẫn xảy ra trong game dù dùng CoreCLR
CoreCLR GC tốt hơn Mono đáng kể, nhưng game dev vẫn phải cẩn thận với allocation. Object pooling, Span<T>, stackalloc, và NativeArray vẫn là best practice quan trọng. CoreCLR giảm tần suất và thời gian GC pause, không loại bỏ hoàn toàn.

Cách Viết C# — Khác Nhau Như Thế Nào?

Phần lớn C# code chạy được trên cả 2 runtime không cần sửa. Nhưng CoreCLR mở ra nhiều pattern mới mà Mono không support hoặc support kém.

Zero-allocation với Span<T>
Mono
Alloc trên heap
// Mono — phải alloc array mới trên heap
public Vector3[] GetPositions(int count) {
    // Mỗi lần gọi = 1 GC allocation
    var result = new Vector3[count];
    for (int i = 0; i < count; i++)
        result[i] = GetPosition(i);
    return result; // boxing + heap alloc
}

// Substring tạo ra string mới trên heap
string name = fullName.Substring(0, 10);
CoreCLR
Zero allocation
// CoreCLR — Span<T> không alloc heap
public void GetPositions(Span<Vector3> dest) {
    // Caller cấp buffer, không alloc mới
    for (int i = 0; i < dest.Length; i++)
        dest[i] = GetPosition(i);
    // Không return, không GC pressure
}

// ReadOnlySpan — zero alloc string slice
ReadOnlySpan<char> name = fullName.AsSpan(0, 10);
Modern C# — Record Types, Init Properties
Mono / C# 9 partial
Không support
// Mono — phải viết boilerplate thủ công
public class PlayerConfig {
    public string Name { get; }
    public int    Level { get; }
    
    public PlayerConfig(string name, int level) {
        Name = name; Level = level;
    }
    // Cần tự implement Equals, GetHashCode...
}

// record không available trên Mono cũ
// init-only setter không có
CoreCLR / C# 12
Đầy đủ
// CoreCLR — record tự động Equals, Hash, Clone
public record PlayerConfig(
    string Name,
    int    Level
);

// Init-only property
public class Config {
    public string Id { get; init; }
}

// Pattern matching nâng cao
var msg = player switch {
    { Level: > 50 } => "Expert",
    { Level: > 20 } => "Mid",
    _ => "Beginner"
};
Native Interop — P/Invoke vs LibraryImport
Mono — P/Invoke
Runtime marshalling
// Mono — P/Invoke với runtime marshalling
// Overhead: type check + marshal tại runtime
[DllImport("__Internal")]
public static extern void showMainActivity(
    string color  // marshalled tại runtime
);

// String phải được pin/marshal
// Không có source generation
// Overhead cao hơn mỗi lần gọi
CoreCLR — LibraryImport
Source generated
// CoreCLR — source-generated P/Invoke
// Zero overhead: marshalling code tạo lúc compile
[LibraryImport("__Internal")]
[UnmanagedCallConv(...)]
public static partial void showMainActivity(
    [MarshalAs(UnmanagedType.LPUTF8Str)]
    string color
);

// Tương đương P/Invoke nhưng không overhead
// Compiler tạo marshalling code = static dispatch

Recommendation cho GaaP App SDK
Với một SDK ship cho nhiều game client trên Unity 6+, chọn CoreCLR. Lý do: performance tốt hơn đáng kể khi xử lý video/stream, GC pause thấp hơn giúp tránh frame spike, C# 12 features giúp viết code sạch hơn (record, Span<T>, pattern matching), và CoreCLR là hướng Unity đang đầu tư — plugin tương thích CoreCLR sẽ không bị kẹt khi Mono bị deprecated.
